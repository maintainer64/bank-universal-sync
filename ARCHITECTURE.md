# Архитектура и как дописывать код

Документ для людей и LLM: как устроен проект, куда добавлять новое и что в коде
стоит упростить.

## Что это

Chrome-расширение (Manifest V3, SolidJS + Vite + Tailwind), которое вытаскивает
данные из личных кабинетов банков/магазинов/брокера и заливает их в **Sure**
(self-hosted финансовый учёт), **Firefly III** или выгружает в CSV.

## Карта репозитория

```
src/
├─ index.tsx / app/app.tsx       # точка входа, hash-роутинг (shared/routing.ts)
├─ service-worker.js             # DNR-правила подмены Origin + прокси fetch
├─ pages/
│  ├─ services/                  # список сервисов (вкладки) + карточка сервиса
│  ├─ sync/sync-page.tsx         # отдельное окно синхронизации с прогрессом
│  ├─ settings/                  # настройки, импорт/экспорт JSON
│  └─ onboarding/                # пошаговый мастер первого запуска
├─ shared/
│  ├─ providers/
│  │  ├─ base.ts                 # ВСЕ доменные типы и интерфейсы (главный файл)
│  │  ├─ registry.ts             # список провайдеров + категории (вкладки)
│  │  ├─ sources/*.ts            # ОТКУДА берём данные (банки, магазины, брокер)
│  │  └─ services/*              # КУДА заливаем (Sure, Firefly, CSV)
│  ├─ sw-fetch.ts                # fetch через service worker (обход CORS)
│  └─ hooks/                     # useUniversalStorage, useServices, useActiveTabUrl
└─ components/ui/                # button, input, label, card, collapsible, progress
test/                            # интеграционные тесты против живого Sure в docker
scripts/                         # одноразовые утилиты (простановка категорий по CSV)
```

## Две главные абстракции

**Источник (`sources/*`)** реализует `ProviderAny` из `base.ts`. Все методы, кроме
`getName/getIcon/getUrl`, необязательные — провайдер объявляет только то, что умеет:

| Метод | Назначение |
|---|---|
| `getKind()` | категория: `bank` / `investment` / `shop` (вкладка в UI) |
| `prepare()` | подготовка перед выгрузкой (у Яндекса — поиск operationId в webpack-бандле) |
| `getAccounts()` | счета |
| `getTransactions()` | операции |
| `getTrades()` | сделки по бумагам (инвестиции) |
| `getProducts()` | заказы (магазины) |

**Приёмник (`services/*`)** реализует `ProviderSync` (`createAccountsIfNotExists`,
`createTransactionsIfNotExists`, опционально `createTradesIfNotExists`) и/или
`ProviderFormatCSV`.

UI и синк построены на проверке наличия методов (`if (provider.getTrades && ...)`),
поэтому новый источник ничего не ломает.

## Как добавить новый источник

1. Создать `src/shared/providers/sources/<name>.ts`, экспортировать объект `ProviderAny`.
2. Сеть — только через `swFetch` (`shared/sw-fetch.ts`): запрос уходит из service
   worker, поэтому не упирается в CORS и получает куки сайта.
3. Если сервису нужна подмена заголовка `Origin` — добавить правило в `DNR_RULES`
   в `src/service-worker.js`.
4. Зарегистрировать в `registry.ts` (там же он попадёт в нужную вкладку по `getKind`).
5. Идентификатор счёта: `institution_domain = "<префикс>_<id счёта у источника>"` —
   по нему сопоставляются счета в Sure. Транзакции: `external_id` + `source`
   (у Sure это ключ идемпотентности, повторный синк не создаёт дублей).

Аутентификация бывает двух видов: **по кукам активной вкладки** (банки, магазины —
`getCookieByName(name, url)`) и **по токену из настроек** (Т-Инвестиции). Токен
приходит в `ProviderParams`.

## Проверка

```bash
pnpm lint && pnpm build          # быстрые ворота
docker compose -f test/compose.sure.yml up -d      # живой Sure
docker compose -f test/compose.sure.yml exec -T -e SEED_API_KEY=... web bin/rails runner - < test/sure-seed.rb
pnpm test:integration            # тесты против него
```

Тесты гоняют **реальный клиент расширения** против настоящего Sure — именно так были
пойманы баги с форматом сумм (`"-$12.00"`), holdings-снимками по датам и дедупом сделок.

# Что стоит отрефакторить

Список по убыванию пользы. Всё — наблюдения из реальной работы с кодом, а не
абстрактные советы.

### 1. ~~Настройки размазаны по строковым литералам~~ — СДЕЛАНО

Теперь есть `src/shared/settings.ts` — единственное место, где заводится настройка:

```ts
export const SETTINGS = {
  'sure-url':           {default: '', label: 'Адрес Sure'},
  'tbank-invest-token': {default: '', label: 'Токен T-Invest OpenAPI', secret: true},
  'onboarding-completed': {default: false, label: '…', exportable: false},
} as const;
```

- `useSetting('ключ')` — типизированный доступ, значение по умолчанию берётся из
  схемы (раньше дублировалось в каждом файле).
- Импорт/экспорт JSON генерируется из `EXPORTABLE_KEYS` — новая настройка
  подхватывается автоматически, руками ничего дописывать не нужно.
- Инварианты схемы закреплены в `test/settings.test.ts`.

Осталось: шаги онбординга всё ещё описаны вручную (там своя вёрстка на шаг),
их можно генерировать из схемы отдельным заходом.

### 2. ~~`ProviderParams` растёт под каждый источник~~ — СДЕЛАНО

Поля конкретных провайдеров убраны из общего типа. Теперь:

```ts
// base.ts
config?: Record<string, string>          // значения нужных провайдеру настроек
getConfigKeys?(): SettingKey[];          // какие именно настройки ему нужны

// провайдер
getConfigKeys: () => ["tbank-invest-token", "tbank-invest-api-url"],
const token = params.config?.["tbank-invest-token"];
```

Страница не знает, какая настройка какому источнику принадлежит: она зовёт
`useSettingsSnapshot()(provider.getConfigKeys?.())`. Новый источник с токеном
не требует правок ни в `ProviderParams`, ни в `SyncRequest`.

### 3. ~~Дублирование сборки `params`~~ — СДЕЛАНО

В `service-details-page.tsx` один `buildParams()` вместо пяти копий объекта.

### 4. Провайдеры не тестируются

`sources/*` дёргают `swFetch` напрямую, поэтому юнит-тестов нет вообще — ошибки
ловятся только руками в браузере.

**Что сделать:** принимать fetch как зависимость (как уже сделано в
`SureInternalApi`: `options.fetch ?? globalThis.fetch`). Тогда можно прогонять
провайдер на сохранённых JSON-фикстурах ответов банка.

### 5. Ответы банков не типизированы

В `sources/*` десятки `any` (`page?.data?.getTransactionsFeedView?.items`). Опечатка
в пути к полю не ловится компилятором и проявляется пустой выгрузкой.

**Что сделать:** описать интерфейсы ответов рядом с провайдером (как сделано в
`tbank_investment.ts`: `InvestAccount`, `InvestOperation`).

### 6. Молчаливые пустые результаты вместо ошибок

Провайдеры возвращают `[[], {}]`, если нет куки/токена. Пользователь видит «синк
прошёл, данных нет» вместо понятной ошибки — ровно так выглядела поломка Яндекса
после переезда синка в отдельное окно.

**Что сделать:** бросать исключение с внятным текстом; окно синка уже умеет его
показывать.

### 7. Мелочи

- `services/sure/intex.ts` — опечатка в имени, должно быть `index.ts`.
- `pnpm lint` проверяет только `src/**`, а pre-commit (lint-staged) — все staged
  `.ts`, включая корневые конфиги. Из-за этого коммит падает на файле, который
  `pnpm lint` считает чистым. Стоит выровнять области проверки.
- `yandex_bank_transactions.ts` (307 строк) стоит разбить: HTTP-слой, парсинг
  бандла, маппинг в доменные типы.
- В `getTransactions` Яндекса внутри повторно вызывается `getAccounts`, хотя синк
  уже получил счета шагом раньше — лишний круг запросов.

### Архитектурный долг: internal-клиент Sure

`SureInternalApi` создаёт счета через веб-форму (CSRF + `DOMParser`), потому что в
API Sure нет создания счетов. Отсюда: зависимость от браузерного DOM (в Node-тестах
не работает — счета в тестах заводятся сидом), хрупкость к вёрстке и та самая
история с подменой `Origin` на nginx. Если в Sure появится API создания счетов —
этот слой надо удалить первым.
